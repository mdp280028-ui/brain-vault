## Fix StackExchange 000000 http_status fetch failures wiping 4 tags for 56 calls/week

**Score:** 4.0 · **Build:** 2.0h · **Impact:** `medium` · **Slug:** `stackexchange-fetch-null-status-fix`

**Signal:** `diagnose.e_repeated_failures_14d[2] (stackexchange_fetch_failed, n=386, http_status:000000)`

stackexchange_fetch_failed logged 386 times in 14 days (diagnose.e_repeated_failures_14d[2]), with the top offenders being asbestos (56), hazardous-materials (56), insulation (56), and demolition (54) in the 7-day window (diagnose.a_top_actions_7d). Every failure example payload shows http_status:"000000" — not a real HTTP status code, which means the request never completed at the transport layer (DNS failure, TLS handshake failure, or connection refused/reset before a response was read). This is not a rate-limit issue; StackExchange's documented throttle returns HTTP 429 with a real status code, not all-zeros. The all-zero status is a curl/transport-layer signature, meaning the current fetch script silently swallows the underlying curl exit code and logs a placeholder instead of the real failure reason.

First action: read the stackexchange fetch script, find where http_status is captured (likely `curl -w '%{http_code}'` on a failed connection returning 000), and log `curl`'s actual exit code plus stderr into the audit_log payload instead of the current placeholder. This turns an unactionable signal into a diagnosable one without changing fetch behavior.

This affects 4 of the poll source's tags every single day, all 168 stackexchange_poll_completed runs still "complete" (diagnose.a_top_actions_7d) despite embedding 56 silent per-tag failures each — meaning the poll's own success signal is misleading. Estimated ~1 recurring transport failure per tag per day, unknown root cause since the failure reason is currently invisible; fixing the logging is the prerequisite to ever fixing the transport issue itself.

If this is wrong: the transport failure might be an intentional/expected block (e.g., IP-based throttling from repeated fetches) rather than a bug, in which case the added logging reveals a wontfix and the only value is the diagnostic clarity itself — a few hours spent for information, not a direct fix.

## Log curl exit code on notify_failed transport_failed_after_3_tries events

**Score:** 4.0 · **Build:** 2.0h · **Impact:** `medium` · **Slug:** `notify-failed-transport-diagnostic`

**Signal:** `diagnose.e_repeated_failures_14d[4] (notify_failed, n=290, http_code:000, transport_failed_after_3_tries)`

notify_failed fired 148 times in 7 days and 290 times in 14 days (diagnose.a_top_actions_7d, diagnose.e_repeated_failures_14d[4]), with the example payload showing http_code:"000", error:"transport_failed_after_3_tries". This is the operator's own Telegram notification pipeline failing at the transport layer before Telegram's API is even reached — the same class of blind spot as the StackExchange 000-status failures, but on the channel the operator depends on to hear about everything else going wrong. A notification pipeline that fails silently at this rate (roughly 20/day) risks masking a real incident precisely when the operator needs to be told about it.

First action: locate the Telegram notify script's retry loop (3 tries, per the payload), and on final failure after exhausting retries, append the curl exit code and a truncated stderr line to the audit_log payload instead of the current generic "transport_failed_after_3_tries" string. No retry-count or backoff changes — purely widening what gets logged on the terminal failure.

Structural risk being mitigated: a real production incident (e.g. a synth failure, a cost-cap trip) firing during a window where the notify transport is also down means the operator gets zero signal for that incident. Frequency: notify_failed already occurs ~20x/day per current data, so the overlap risk is not hypothetical — it is a recurring condition. Impact if a real alert is lost during one of these windows: operator misses a same-day-actionable issue, discovered only via manual audit_log review, unknown exact recovery cost per incident but consistent with the ~30-60min pattern seen in other diagnose-driven fixes this cycle.

If this is wrong: the 000 code may just be a benign local network blip (Mac Mini WiFi/DNS hiccup) that self-heals via existing retries, in which case the added logging confirms a non-issue and the 2-hour build was spent on confirmation rather than a fix.
