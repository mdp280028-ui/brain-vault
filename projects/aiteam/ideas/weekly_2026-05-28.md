## Add PID liveness check to lockfile logic to prevent permanent silent agent stoppage

**Score:** 4.0 · **Build:** 2.0h · **Impact:** `medium` · **Slug:** `stale-lockfile-pid-guard`

**Signal:** `diagnose.e_repeated_failures_14d[2] (lock_skip: 6, example payload {"existing_pid": 69401}) + search result unix.stackexchange.com/questions/12815 (PID lock file validation idiom)`

The `e_repeated_failures_14d` table shows 6 `lock_skip` events in 14 days with payload `{'existing_pid': 69401}`. The current lockfile pattern writes a PID on entry and skips all subsequent invocations if the file exists. It does not verify whether the recorded PID is still alive. When a cron-invoked agent crashes mid-run (OOM kill, timeout, SIGKILL from the OS), the lockfile remains with a now-dead PID. Every subsequent cron invocation silently skips. The agent produces no output and writes no failure entry to audit_log. No Telegram alert fires. The failure is invisible until the operator notices the silence.

Audit all lockfile acquire paths in `~/agents/lib/` and key agent entry scripts. For each path, add a PID liveness check with `kill -0 $pid 2>/dev/null`: if the stored PID is not running, remove the stale lockfile with `rm -f` and proceed normally. The `unix.stackexchange.com/questions/12815` search result confirms `kill -0` as the POSIX-standard idiom for this check. The fix is a two-line addition per lock-acquire site.

Track B: Failure mode — agent crashes mid-run, leaving a stale lockfile that permanently blocks all subsequent cron invocations. The agent silently stops drafting, curating, or checking watchdog state. Watchdog may not detect this condition if the agent's heartbeat is written at start rather than end. Frequency: 6 lock_skip events in 14 days; how many are stale vs. legitimate concurrent runs is currently unknown — the payload shows the same PID 69401 referenced but does not disambiguate. Impact: undetected agent stoppage for potentially days; recovery requires the operator to manually find and delete the stale lockfile, approximately 0.5-1 hour per incident depending on which agent is affected.

The PID recycling race is the principal downside: if the OS reuses the dead agent's PID for an unrelated process in the window between the liveness check and the lockfile removal, `kill -0` will return success against the wrong process and the stale lockfile will be incorrectly retained. This scenario is negligible on a single-user Mac Mini with low process churn but is real in theory.
