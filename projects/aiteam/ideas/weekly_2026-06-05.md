## Auto-restart content-discovery agents when drafter queue runs dry

**Score:** 4.200000000000000177 · **Build:** 2.0h · **Impact:** `medium` · **Slug:** `content-discovery-auto-restart`

**Signal:** `diagnose.d_dead_agents_30d_no_7d (research-opportunity-extract, research-opportunity-triage) + diagnose.f_drafter_queue (asbestos_pending=0, ssg_pending=0)`

research-opportunity-extract and research-opportunity-triage have had no audit_log activity in the past 7 days (diagnose.d_dead_agents_30d_no_7d). Both sites' drafter queues are empty (diagnose.f_drafter_queue: asbestos_pending=0, ssg_pending=0). Without new slug candidates feeding the drafter, no content is drafted, no content ships, and the path to $200/month stalls indefinitely. The pending content-pipeline-stall-alert proposal already in the queue surfaces this break as an operator notification — this proposal adds a self-healing recovery layer with a different mechanism.

A content-discovery-watchdog.sh script, installed as a cron at 08:00 daily, checks three conditions before firing: (1) both sites' drafter_pending==0 per live SQLite query against aiteam.db, (2) research-opportunity-extract last audit_log entry is older than 24 hours, (3) no halt flag present and LLM_SPAWN_ENABLED is not false. When all three hold, it calls run_agent.sh research-opportunity-extract once and logs a content_discovery_autostart event via log_to_audit.sh. It does not loop. One restart attempt per 24-hour window maximum.

This complements rather than replaces the stall-alert: the alert fires immediately on detection; the auto-restart fires as a fallback if the operator does not manually intervene within 24 hours. Together they reduce mean time-to-recovery from operator-response-time to at most 24 hours.

Failure mode prevented: silent multi-day stall of the content-discovery pipeline with no self-healing mechanism. Frequency: current stall is 7+ days ongoing; prior stall frequency unknown — no prior auto-restart incidents on record. Impact if stall persists unresolved: drafter queue stays at 0, content publishing halts, fleet generates $0 revenue until operator manually intervenes. Downside: if research-opportunity-extract is failing for a repeating internal reason, auto-restart re-triggers the same failure daily and adds audit_log noise without resolving root cause.

## Add lowfat CLI output filter as PreToolUse Bash hook to shell agent

**Score:** 4.200000000000000177 · **Build:** 1.0h · **Impact:** `medium` · **Slug:** `lowfat-shell-agent-hook`

**Signal:** `diagnose.b_top_cost_agents_7d[2] (shell, $4.93/7d, 127 calls) + diagnose.c_cost_per_call_delta_14d[3] (shell CPC +45%, from $0.026 to $0.039) + scan.hackernews[3] (lowfat, 91.8% token reduction via PreToolUse Bash hook)`

The shell agent ran 127 calls in 7 days at an estimated $4.93 total ($0.039/call), ranking 3rd in fleet cost. Cost per call rose 45% over 14 days (diagnose.c[3]: from $0.026 to $0.039). The scan surfaced lowfat, an open-source CLI filter that acts as a Claude Code PreToolUse Bash hook, compressing verbose shell output (git log, find, docker ps, ls -la) by a measured 91.8% before it reaches the LLM. Adding it to the shell agent's .claude/settings.json is a single JSON hook block — the same settings file that already holds the Bash allow-rules.

Shell agent token cost breaks into system prompt (fixed), command output (variable), and model response. Lowfat only compresses the command output portion. Conservative estimate: if command output is 25-40% of total input tokens and lowfat removes 80% of that, per-call savings is $0.008-$0.012. At 127 calls/7d that is $1.00-$1.50/week, or $4-7/month against the daily soft cap. The actual delta is measurable in audit_log token counts after 3 runs; `lowfat stats` also tracks lifetime savings natively.

Start at `--level lite` (less aggressive compression) to avoid stripping exit-code signals the agent reads to branch on failure. Promote to default level only after confirming error-detection paths fire correctly on a known-failure scenario in a controlled run.

Downside: if lowfat compresses command output in a way that hides error signals the shell agent branches on, it could cause the agent to silently mishandle tasks or miss failures. The lite-first approach limits this risk, but the first post-deployment session should be reviewed manually for any suppressed failure signal.

## Diagnose ssg-auditor $0.32/call context to cut $47/month fleet spend

**Signal:** `diagnose.b_top_cost_agents_7d[0] (ssg-auditor, $10.94/7d, 34 calls, $0.32/call)`

ssg-auditor is the fleet's highest absolute cost agent this week: 34 calls at $10.94 total ($0.32/call), projecting to ~$47/month. This exceeds ssg-writer ($6.09/week, 24 calls, $0.25/call), meaning the auditor costs 28% more per call than the writer. Auditing a fixed-content page against a spec should require less output than drafting it from scratch; the inversion is a signal of context overload, not model depth. The most common cause is passing the full system prompt + full article body + full template spec in a single unbounded context rather than separating static spec from live content.

The investigation path: read ssg-auditor's run script and measure total chars passed per invocation. If the system prompt carries a large static audit spec (>30K chars), the immediate action is to move that spec to a prompt-cached prefix so it is not rebilled per call. File-tiering approaches that pass only relevant context have achieved 60-94% cost reduction on similar auditing workloads (search: reddit.com/r/ClaudeAI/comments/1qp9ve9). Pending idea 'asbestos-auditor-cost-root-cause' targets a different agent with a different content schema; this investigation is independent.

Track A: Current cost ~$47/month. A 50% context reduction via prompt caching on static spec content saves ~$23/month. A 75% reduction (achievable if the spec dominates context) saves ~$35/month. Cost to investigate: 2-3h regardless of outcome.

If the investigation finds the full spec is legitimately required on every call and no caching boundary exists, the cost is 2-3h of investigation time with no code change produced.

## Tombstone assignment slugs after 3 consecutive validation failures

**Score:** 3.799999999999999823 · **Build:** 3.0h · **Impact:** `medium` · **Slug:** `assignment-draft-validation-tombstone`

**Signal:** `diagnose.e_repeated_failures_14d[4] (assignment_draft_failed, n=10, reason=validation, site=asbestos, slug=3m-respirator)`

The assignment drafter logged 10 validation failures in 14 days (diagnose.e_repeated_failures_14d[4], reason='validation', site='asbestos'). The example slug — 3m-respirator — also appears in config_synth_failed and skipped events with reason='synth_failed', indicating the same slug is cycling through multiple pipeline stages and failing repeatedly. No ceiling on retries exists: a slug that fails validation re-enters the queue on the next drafter tick, consuming queue capacity and (if validation fires post-Sonnet) LLM budget each cycle.

The fix is a consecutive-failure counter on the slug queue — either a dedicated column or a lightweight tombstone table — that increments on each validation failure and halts re-queuing after 3 consecutive failures, writing a `slug_tombstoned` audit event for operator review. The config-synth-stuck-slug-tombstone idea (pending) implements this pattern for config synthesis; this applies it one phase earlier at assignment draft validation, covering a different failure mode in a different script.

The failure mode prevented is an indefinitely-cycling slug that occupies a queue slot and generates orphaned drafter_tick_started rows without ever producing a shippable article. At 5 failures per week, the throughput tax is ongoing and directly competes with the $200/month revenue target: at audit_guide's current 33% pass rate (diagnose.g_verdict_pass_rate_14d), every unshippable slug cycling in the queue displaces a slug that could pass. Estimated frequency: 5 validation failures per week currently. Estimated impact if unaddressed: queue partially occupied by structurally unshippable slugs for the lifetime of each stuck slug, constraining the content pipeline that is the primary path to revenue.

If this is wrong: validation failures might be transient — slug metadata temporarily missing, dependency not yet resolved — in which case a 3-failure tombstone permanently suppresses a slug that would eventually pass. Mitigation: make the threshold configurable and emit the tombstone event so the operator can manually clear it.

## Add EXIT traps to loop, domain-hunter, and rebuild-tick cron scripts

**Score:** 3.600000000000000088 · **Build:** 2.0h · **Impact:** `medium` · **Slug:** `non-drafter-exit-trap-sweep`
