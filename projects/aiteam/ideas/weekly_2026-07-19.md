## Fix empty http_code/error fields on notify_failed so Telegram outages are diagnosable

**Score:** 3.799999999999999823 · **Build:** 1.5h · **Impact:** `medium` · **Slug:** `notify-failed-empty-http-code-fix`

**Signal:** `diagnose.e_repeated_failures_14d[2] (notify_failed, n=310, http_code=000)`

notify_failed logged 159 times in 7 days and 310 times in 14 days (diagnose.a_top_actions_7d, diagnose.e_repeated_failures_14d[2]), with the example payload showing http_code:"000", ok:"false", error:"transport_failed_after_3_tries". An http_code of "000" means curl never got a response (DNS failure, connection refused, or timeout before the socket opened), but the current payload does not distinguish those cases or record curl's actual exit code, so every one of these 310 events is logged identically and none of them is actionable without re-running curl manually.

The fix adds `curl_exit_code=$?` and `curl_errno` to the notify_failed payload in the Telegram send wrapper, alongside a DNS-resolvable check (`getent hosts api.telegram.org` or equivalent) so the log can distinguish "no network" from "Telegram API down" from "malformed request." This is a single wrapper-script edit plus a payload field addition, no new files.

Prevents continued blind spot on Telegram outages, which is the operator's only out-of-band channel for weekly_run_failed and watchdog_alert. Frequency: ~22 notify_failed/day at current rate (310/14d), each one currently unactionable. Impact if unfixed: the operator has no way to tell a real Telegram/network outage from a transient blip, so a genuine multi-hour Telegram outage (during which fail-fast chain halts would go unreported) looks identical to noise in the logs — recovery time for such an outage would be however long it takes the operator to notice manually, likely hours, since this is the escalation channel itself.

Downside: if wrong, this is a few extra fields in a JSON payload with zero behavior change — worst case is slightly noisier audit_log rows that still don't pinpoint the cause, at near-zero cost.

## Add per-tag circuit breaker for StackExchange fetches after N consecutive http_status=000000 failures

**Score:** 3.600000000000000088 · **Build:** 2.0h · **Impact:** `medium` · **Slug:** `stackexchange-null-status-circuit-breaker`

**Signal:** `diagnose.e_repeated_failures_14d[4] (stackexchange_fetch_failed, n=198, http_status=000000)`

stackexchange_fetch_failed logged 198 times in the diagnose e_repeated_failures_14d table this run, with http_status "000000" (a malformed/impossible HTTP status, distinct from a real 4xx/5xx) across tags including hazardous-materials. A prior idea (stackexchange-fetch-null-status-fix, pending, proposed 2026-07-10) targets diagnosing and fixing the root cause of the 000000 status itself. This idea is a different mechanism: regardless of root cause, add a circuit breaker that stops hitting a tag after 5 consecutive 000000 failures within a 24h window, sleeping that tag's poll for 6 hours before retry, rather than re-attempting on every poll_tick (every ~1h per diagnose.a_top_actions_7d showing poll_tick_started n=163/7d).

This is not a duplicate: the prior idea fixes the cause (why status is empty), this idea bounds the blast radius (stop burning poll cycles on a known-broken tag) independent of whether the cause is ever fixed. Both can ship; this one is a safety net if the root-cause fix stalls or a new tag breaks the same way later.

Saves operator time by removing a permanently-noisy log source: at 198 failures/14d across just the sampled tags, extrapolated across all polled tags this is a meaningful fraction of the 149 poll_source_skipped/reddit-style noise the operator already has to filter past to find real signal. No dollar cost since StackExchange API calls are free, so this is a Track B signal-quality argument, not cost savings — estimated ~1-2 operator minutes/day currently spent mentally filtering these rows during log review, roughly 8-10 hours/year.

Downside: if the underlying issue is transient (e.g., a temporary StackExchange outage) rather than tag-specific, the circuit breaker could suppress a tag from being polled during a period when it would have recovered, delaying discovery of new content by up to 6 hours — an acceptable tradeoff given the tag has already failed 5 consecutive times.
