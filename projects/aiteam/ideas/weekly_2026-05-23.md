## Split oversized Telegram notifications into numbered multi-part sends

**Score:** 4.200000000000000177 · **Build:** 1.0h · **Impact:** `medium` · **Slug:** `telegram-message-chunker`

**Signal:** `e_repeated_failures_14d[3] (notify_failed, HTTP 400, 'message is too long', 4693 bytes, 3 occurrences in 14 days)`

The Telegram notify pipeline fails silently when payloads exceed 4,096 bytes: Telegram returns HTTP 400, the notification is dropped, and the operator receives nothing. Three notifications were lost this way in the past 14 days (bytes: 4,693 in the confirmed example — just 15% over the hard limit). The fix is a byte-length check before each curl send in the notify script: if the message exceeds 3,800 bytes, split on newline boundaries into N chunks and send each sequentially with a '(1/N)' header prefix.

Track B: failure mode = critical operator notification silently dropped; frequency = 3 lost notifications in 14 days (~1.5/week, confirmed from audit_log); impact = operator remains blind to the triggering event until the next manual log inspection, which in a Telegram-primary alerting system can be hours. The current state is zero messages delivered — any chunked delivery is strictly better.

If this is wrong — if multi-part messages create reading confusion on a phone — the cost is minor UX friction. The implementation adds no new dependencies and no LLM calls.

## Add EXIT trap to drafter tick script for completion and failure logging

**Score:** 4.0 · **Build:** 2.0h · **Impact:** `medium` · **Slug:** `drafter-tick-completion-trap`

**Signal:** `h_orphaned_starts_7d[0] (drafter_tick_started, 162 orphan events with zero corresponding completion events in 7 days)`

The drafter cron fires approximately 23 times per day and logs drafter_tick_started on entry. In the past 7 days, 162 drafter_tick_started events were logged with zero corresponding completion or failure events. Every single drafter tick is an operational black box: diagnose.sh cannot distinguish 'healthy idle tick with no work' from 'tick crashed silently' from 'tick was locked out' from 'tick processed slugs and exited cleanly.' With 3 slugs currently pending in the drafter queue (f_drafter_queue) and content production at $0, not knowing whether the drafter is stuck or merely idle is a direct blocker on diagnosis.

The fix: add a bash EXIT trap at the top of the drafter tick script that fires on every exit path. The trap reads $? and maps it to a reason category — no_work (exit 0, zero slugs processed), completed (exit 0, slugs processed), lock_skip (exit triggered by PID lock guard), error (exit non-zero) — then calls log_to_audit.sh with action=drafter_tick_completed and payload containing reason, exit_code, and slugs_processed.

Track B: failure mode = drafter exits abnormally or silently skips pending work with no audit trail, leaving the operator unable to distinguish a healthy system from a broken one; frequency = 162 unresolved orphan starts in 7 days — this is happening on every single tick right now, not hypothetically; impact = root-cause diagnosis of a stalled content queue requires manual log-grep instead of a diagnose.sh query, estimated ~1 hour investigative overhead per incident based on the established pattern for orphaned-start debugging.

If this is wrong — if an existing EXIT trap in the drafter tick conflicts — bash trap semantics (last-writer wins) will silently drop the new handler. Read the script for existing traps before adding; compose if necessary.

## Add retry-with-backoff to assignment drafter's Sonnet call

**Score:** 4.0 · **Build:** 2.0h · **Impact:** `medium` · **Slug:** `drafter-sonnet-retry`

**Signal:** `e_repeated_failures_14d[1] (assignment_draft_failed, reason: sonnet, 7 occurrences in 14 days)`

The assignment drafter makes one Sonnet call per slug and exits permanently on failure. Seven drafts failed in 14 days with reason 'sonnet', averaging one failure every two days. Each failure leaves the slug in the pending queue with no retry and no escalation. The drafter queue currently holds 3 pending slugs; at the observed failure rate, at least one of those slugs has already failed at least once without advancing.

The fix: wrap the Sonnet call in a retry loop (3 attempts, exponential backoff: 30s, 60s, 120s). Before entering the retry loop, check LLM_SPAWN_ENABLED and SYSTEM_PAUSED — if either kill switch is active, log drafter_sonnet_blocked and exit immediately without retrying. On final failure after 3 attempts, log drafter_sonnet_failed with {attempt_count: 3, final_error_code: N}.

Track B: failure mode = slug permanently stuck in pending queue after a single transient Sonnet error; frequency = 7 in 14 days (~0.5/day, confirmed from audit_log); impact = each stuck slug is content not shipped, which is a direct block on the $0 → $200/month mission bar. With 3 slugs pending and a 0.5/day failure rate, the queue is advancing slower than it is failing.

If this is wrong — if the Sonnet failures are structural rather than transient (account issue, model unavailable) — the retry loop adds up to 3 minutes of latency before the failure is logged. The kill-switch pre-flight check limits this for the most common structural case.

## Log structured rejection category on editor-v1 failures

**Score:** 4.0 · **Build:** 2.0h · **Impact:** `medium` · **Slug:** `editor-rejection-reason-tagger`

**Signal:** `g_verdict_pass_rate_14d[0] (editor-v1, 1/4 passed, 25% pass rate over 14 days); corroborating: d_dead_agents_30d_no_7d[5] (editor inactive last 7 days)`

editor-v1 passed 1 of 4 verdicts in the 14-day window (25% pass rate). The editor logs a binary pass/fail verdict to audit_log but does not log WHY it rejected. Without rejection-reason data, the operator cannot determine whether the 75% rejection rate reflects miscalibration — blocking good content in violation of Q1 policy — or correct quality enforcement on genuinely weak drafts. The editor also appears in d_dead_agents_30d_no_7d, meaning it has been inactive in the last 7 days; the 4 verdicts are from the 10-14 day window. If the drafter's Sonnet failures (e_repeated_failures_14d[1]) are resolved and drafts begin flowing again, the operator will face the same calibration ambiguity at higher volume.

The fix: in the editor script's post-verdict step, parse the raw Sonnet response for the first stated failure criterion and append a rejection_reason field to the editor_verdict audit_log payload. Define a fixed category set before implementation (e.g., thin_content, structure, word_count, duplicate_angle, factual_error) so the tagger maps to structured values rather than free text.

Track A: saves ~1 operator hour/week currently spent manually opening raw editor responses to understand rejection patterns. At the current 4-verdict sample, there is no structured way to separate calibration drift from real quality failures without reading each response individually. Once logging exists, a simple diagnose.sh GROUP BY rejection_reason query surfaces the distribution within one or two editor runs.

If this is wrong — if the rejection-reason categories are too coarse or the parser mis-tags Sonnet's response — the binary pass/fail verdict is unaffected. Only the diagnostic field is wrong, and wrong tags are discoverable by comparing to raw responses.

## Deduplicate slug_already_shipped audit_log writes to 1 per slug per day

**Score:** 3.799999999999999823 · **Build:** 1.0h · **Impact:** `medium` · **Slug:** `ship-to-site-skip-dedup`

**Signal:** `e_repeated_failures_14d[0] (ship_to_site_skipped, 1142 occurrences in 14 days, all reason: slug_already_shipped; corroborating: a_top_actions_7d[7-8] showing 16 skips each for asbestos-shingles-guide and chrysotile)`

The ship-to-site agent checks every candidate slug against APPROVED_GUIDE_SLUGS on each tick and logs ship_to_site_skipped for already-shipped slugs. In the past 14 days, this generated 1,142 skip events (81.6/day), all with reason slug_already_shipped. This single action type holds the top slot in e_repeated_failures_14d, displacing genuine failure signals. The two confirmed sites contributing to the skip volume (asbestos-shingles-guide, chrysotile) each show 16 skip events visible in a_top_actions_7d, indicating a small set of slugs are being rechecked continuously rather than the list growing.

The fix: in the ship-to-site tick, before writing ship_to_site_skipped to audit_log, execute a SQLite query checking for the same slug+action+site in the last 86,400 seconds. If a matching row exists, skip the write. This reduces the 81.6 writes/day to approximately 5-10 unique-slug writes/day, restoring the repeated_failures diagnose query to its intended purpose of surfacing novel failures.

Track A: saves ~0.5 operator hours/week spent filtering skip-noise from diagnose output. Secondary: reduces the token input burden on future idea-agent synthesis passes, where a 1,142-row slug_already_shipped block displaces signal from diagnose fields e through n. The dedup logic adds one read-only SQLite query per tick per site.

If this is wrong — if the dedup guard fires during a period when APPROVED_GUIDE_SLUGS is corrupted and slugs are being incorrectly skipped — the 24-hour dedup window suppresses repeated-skip events that would otherwise flag the corruption. Mitigate by comparing slug+reason+site in the dedup key, not slug alone.
